# Architecture Decision Records (ADR)

Фиксируем ключевые решения по проекту. Формат: проблема → варианты → решение → обоснование.

---

## ADR-001: Декомпозиция Orchestrator

**Статус:** принято
**Дата:** 2026-03-13

**Проблема:** Orchestrator в исходной архитектуре — God Object (8 компонентов в одном контейнере). Одна точка отказа, нельзя масштабировать отдельные части.

**Решение:** Разделить на независимые сервисы:
- Pipeline Manager — управление состояниями research case
- Task Dispatcher — постановка задач в очередь
- Evaluator — валидация evidence + скоринг

**Обоснование:** Каждый сервис можно масштабировать, деплоить и тестировать независимо.

## ADR-002: Message broker между оркестратором и воркерами

**Статус:** принято
**Дата:** 2026-03-13

**Проблема:** Синхронный запрос-ответ между Orchestrator и агентами. LLM-вызовы долгие (10-60 сек), нет retry, нет параллелизма.

**Решение:** BullMQ + Redis. Dispatcher кладёт задачи в очередь, воркеры забирают асинхронно.

**Обоснование:** Асинхронная обработка, автоматический retry, параллельный запуск агентов, устойчивость к сбоям.

## ADR-003: Минимальный набор агентов для MVP

**Статус:** принято
**Дата:** 2026-03-13

**Проблема:** Спека описывает 30+ агентов. Даже Phase 1 предполагает ~17. Каждый агент = промпт + контракт I/O + тесты + мониторинг.

**Решение:** Phase 1 = 3 агента:
- Researcher — один агент, разные промпты для секций (market, competitors, signals, tech, risk)
- Critic — проверяет quality и evidence
- Scorer — считает score, формирует вердикт

**Обоснование:** Быстрее до первого результата. Масштабировать количество агентов позже, когда ядро стабильно.

## ADR-004: Обратные петли в pipeline

**Статус:** принято
**Дата:** 2026-03-13

**Проблема:** Линейный workflow (Research → Critic → Report). Если Critic находит слабые доказательства — некуда отправить на доработку.

**Решение:** Critic возвращает pass/fail по каждой секции. Fail = задача обратно в очередь Researcher с фидбеком от Critic. Лимит: 2-3 итерации на секцию.

**Обоснование:** Качество research повышается итеративно. Лимит итераций предотвращает зацикливание.

## ADR-005: Воркеры пишут в DB напрямую

**Статус:** принято
**Дата:** 2026-03-13

**Проблема:** Данные (результаты research, evidence, scores) ходят через Orchestrator. Он становится bottleneck при параллельных агентах.

**Решение:** Воркер получает задачу из очереди → LLM-вызов → пишет результат в DB сам → отправляет событие "done" в брокер. Оркестратор работает с состояниями, не с данными.

**Обоснование:** Убирает bottleneck, упрощает оркестратор, снижает нагрузку на сеть.
